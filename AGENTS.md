# 1. 编码前先思考
- 默认使用中文回复，除非我明确要求使用其他语言
- 明确说明假设，不确定就问
- 存在多种解释时，列出方案让我选
- 如果有更简单的方法，直接说出来
- 困惑时停下来，指出不清楚的地方

# 2. 简洁至上
- 不要添加没要求的功能
- 不要为一次性代码创建抽象
- 不要加不必要的“灵活性”或“可配置性”
- 不要为不可能发生的场景做错误处理
- 能用50行写完，绝不写200行

# 3. 精准修改
编辑现有代码时：
- 不要“改进”相邻的代码、注释或格式
- 不要重构没坏的东西
- 匹配现有代码风格
- 发现无关死代码时，先告知，不要擅自删除

当你的改动产生孤儿代码时：
- 删除因你的改动而变得无用的导入/变量/函数
- 不要删除原本就存在的死代码（除非我要求）

# 4. 目标驱动，自己验证
把任务转化为可验证的目标：
- “加验证” → “先写无效输入的测试，再让测试通过”
- “修Bug” → “先写复现Bug的测试，再修复”
- “重构X” → “保证重构前后测试全绿”

多步骤任务，先列计划：
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]

# 5. 项目产品约束

本项目名称为“泉峰AI数据中心管理平台”，定位为本地优先、开源、可扩展的 AI 数据中心机柜资产与运维管理平台。

长期产品定位：
- 做成面向真实数据中心运维场景的成熟开源 AI DCIM 平台
- 不是单纯机柜画图工具，也不是单纯 AI 聊天工具
- 以机柜资产数据为底座，逐步统一机房空间、设备资产、端口连线、监控告警、巡检记录、变更记录和 AI 助手
- 最终成为数据中心日常运维入口，并预留企业级服务化、API 化、插件化能力

第一版优先目标：
- 以专业好看的 2D 机柜图为核心
- 优先满足日常运维可用
- 支持表单录入和简单 Excel 模板导入
- 支持设备搜索、资产查询、机柜/U 位定位
- 支持基础告警状态展示
- 支持 AI 查询真实平台数据
- 3D 视图先做轻量展示，不作为第一版核心成败点

第一版暂不优先：
- 不先做完整权限系统
- 不先做多人协作
- 不先做复杂工单系统
- 不先做完整 Prometheus/卓豪生产接入
- 不先做复杂接线拓扑自动布局
- 不在笔记本安装本地大模型
- 不一开始做插件市场

第一版数据库需要预留端口和连线模型：
- 预留 Port、Connection、DeviceTemplate、ChangeRecord 等基础结构
- 设备详情中可预留端口/连线标签页
- 连线管理页面第一版可以只做基础入口和空状态
- 不在第一版实现完整端口占用、线缆拓扑、交换机接口图和接线表
- 后续版本再正式增强连线管理能力

第一版机房与机柜建模规则：
- 默认机柜高度为 48U
- 数据模型保留 DataCenter + Room 两层
- DataCenter 表示地区、园区或数据中心归属，例如南京、杭州、越南
- Room 表示具体机房或数据中心房间，例如 529数据中心、99数据中心、308数据中心、杭州数据中心、越南C7数据中心
- UI 上可以简化显示为“数据中心/机房”，但底层模型不要合并
- v0.1 示例核心机房为“529数据中心”
- 529数据中心由 2 个华为微模块组成
- 每个华为微模块包含 20 个柜位，布局为 2 排 x 10 列
- 529数据中心合计 40 个柜位
- 不能把 529数据中心的微模块结构硬套到所有机房
- 机房必须支持不同布局类型：微模块布局、简单排柜布局、单柜布局
- 南京99机房、南京308机房、杭州机房、越南C7机房等可以使用简单排柜布局或单柜布局
- 对简单机房，微模块字段可以为空
- 对只有一排机柜或一个机柜的机房，排、列也应允许简化配置
- 第一版预置数据中心/机房名称：529数据中心、99数据中心、308数据中心、杭州数据中心、越南C7数据中心
- 用户在机柜总览中选择实际数据中心/机房名称，不直接选择“微模块总览”或“简单机房总览”
- 布局类型由系统根据数据中心/机房配置自动决定
- 机柜名称按现场实际名称录入，平台不强制固定命名规则
- 示例数据可以使用统一简洁命名，但不能成为系统限制
- Rack 既表示普通服务器机柜，也表示一个机房柜位/基础设施柜体
- Rack 必须支持不同类型：服务器机柜、网络机柜、存储机柜、配线机柜、列头柜、精密空调、UPS/PDU、空柜、其他
- 精密空调、列头柜、配线机柜等特殊柜体默认可作为整柜块显示
- 特殊柜体也要支持切换为 48U 设备视图，以便后续真实资产管理

U 位规则：
- 第一版默认 48U
- U 位编号从下往上，1U 在底部，48U 在顶部
- startU 表示设备底部起始 U 位
- endU = startU + heightU - 1
- 例如 startU=25、heightU=4，则占用 25U-28U
- 设备不得超出 1U-48U 范围
- 同一机柜、同一安装面内不得出现 U 位冲突
- 第一版安装面支持正面、背面，2D 展示优先实现正面

# 6. AI 助手设计约束

AI 助手是全局能力，不作为独立主导航页面。

AI 助手交互形态：
- 通过顶部右侧“AI 助手”入口唤起
- 默认从右侧弹出
- 默认宽度约 420px
- 支持收起、展开、放大、缩小
- 支持扩展到更宽面板或全屏模式
- 支持最小化为悬浮入口
- 支持固定在当前页面右侧
- 在页面切换时尽量保持会话上下文

AI 助手必须能读取当前页面上下文：
- 在机柜总览中，读取当前机房、机柜、设备、告警上下文
- 在资产管理中，读取当前设备列表、筛选条件、选中设备上下文
- 在告警中心中，读取当前告警、关联设备、位置上下文
- 在报表中心中，读取当前统计范围和报表上下文

AI 第一版能力边界：
- v0.1 只做查询型 AI 助手，不做 AI 写入、修改、删除、移动设备
- 支持自然语言查询设备位置、用途、责任人、资产信息、机柜设备清单、告警状态
- 支持生成当前机房或当前页面状态摘要
- 支持问题示例：“哪个服务器在哪里？”、“这台服务器用途是什么？”、“责任人是谁？”、“某个机房/机柜有哪些服务器？”
- AI 回答资产、告警、位置、运行状态时，必须基于平台真实数据，不得编造
- 新增设备草稿、修改设备草稿、AI 调整机柜位置等能力作为后续版本预留，不进入 v0.1 必做范围
- 后续版本如开放 AI 写入能力，必须经过用户确认、权限控制和审计记录

AI 模型配置规则：
- 第一版支持多个模型配置，但同一时间只启用一个默认模型
- AI 助手默认使用当前启用模型
- 支持后续新增、编辑、删除、启用/停用模型配置
- 支持按供应商/协议分类管理模型
- 支持测试模型连接
- 支持从可发现接口拉取可用模型列表，并让用户选择具体模型
- API Key 不得写入项目导出文件
- 外部模型配置需要提醒用户注意敏感数据和内网信息脱敏

第一版模型供应商/协议分类：
- GPUStack，本地公司 GPU 一体机模型，优先支持 qwen3.6-35b
- OpenAI Compatible，通用兼容接口
- DeepSeek
- Gemini
- Ollama
- vLLM
- 其他

每个模型配置至少包含：
- 配置名称
- 供应商/协议类型
- Base URL
- API Key，可为空
- 模型标识 model
- 是否启用
- 请求超时时间
- 最大输出长度
- temperature
- 备注

模型发现规则：
- OpenAI Compatible、GPUStack、vLLM、Ollama 优先通过模型列表接口发现可用模型
- 如果供应商不支持模型发现，允许用户手动填写模型标识
- 发现模型失败时不能阻塞手动配置
- 测试连接必须显示成功/失败原因
- 测试连接只发送简单低风险提示词，不携带真实资产数据

AI 审计与操作审计规则：
- 第一版必须预留 AI 审计功能
- 每次 AI 问答需要记录审计日志，方便后续查询历史
- AI 审计记录至少包含：时间、用户、问题、命中的工具、查询条件、回答摘要、关联设备、关联机房/机柜、模型配置、结果状态
- 第一版 AI 审计只记录查询行为，不记录敏感 API Key
- 审计中心需要支持按时间、关键词、设备、机房、机柜、操作类型搜索
- 后续版本如果 AI 支持新增设备、调整设备、修改配置，必须把修改前、修改后、调用来源、确认人、执行结果写入审计记录
- 外部 AI Agent 未来通过 API 查询或修改平台时，也必须进入同一套审计体系
- 审计记录用于追溯 AI 查询历史、人工操作历史、外部 Agent 调用历史和资产变更历史
- 第一版审计日志入口放在系统设置中，作为“审计日志”标签页
- 后续当 AI 操作、外部 Agent 调用、资产变更、导入导出日志复杂后，再独立为“审计中心”主导航页面

第一版 AI 只读工具集：
- search_devices：按计算机名、IP、业务IP、带外IP、资产编号、SN、用途、责任人搜索设备
- get_device_detail：查看单台设备完整详情
- locate_device：返回设备所在数据中心、机房、微模块、机柜、U 位
- list_rack_devices：查询某个机柜里的设备清单
- list_room_devices：查询某个数据中心/机房里的设备清单
- list_alert_devices：查询告警、离线、维护等状态设备
- summarize_room_status：汇总某个机房设备数量、状态、告警、机柜占用情况

第一版 AI 工具全部只读，不允许修改数据库。

AI 查询回答格式要求：
- 回答必须专业、结构化、可追溯
- 涉及设备时优先展示计算机名、业务IP、用途、责任人、位置、状态
- 涉及位置时展示数据中心/机房、微模块、排、列、机柜、U 位范围
- 涉及机柜时展示机柜名称、机柜类型、设备数量、异常设备数量
- 每次查询回答必须显示数据来源和查询时间
- 数据来源示例：本地资产库、本地告警库、导入批次、后续监控适配器
- 查询时间使用本机当前时间
- 如果查不到数据，要明确说明没有匹配记录，并建议用户检查关键词、IP、资产编号或导入数据

# 7. 技术栈约束

- 路由：Vue Router
- 状态管理：Pinia
- UI 组件库：Naive UI

- 3D 展示：Three.js

- AI 接口：OpenAI Compatible API
- 模型来源：公司 GPU 一体机 qwen3.6-35b，不在笔记本安装本地大模型

架构原则：
- 数据模型先行，页面和 AI 能力围绕统一数据模型建设
- 第一版桌面优先、本地优先，后续预留服务化和 API 化
- 监控系统、AI 模型、导入导出、通知能力都按适配器/插件思路设计
- 不在业务代码中硬编码某一个监控系统或某一个模型服务
- 2D 机柜图是第一版核心能力，性能和清晰度优先
- 3D 视图第一版轻量展示，避免为了炫技拖慢核心功能

# 14. 3D 轻量视图约束

3D 视图第一版定位为“轻量但具备领导展示价值”。

- 能看出 529数据中心的华为微模块结构，以及其他机房的简单机柜布局
- 能显示每个机柜编号或名称
- 能用颜色区分普通机柜、特殊柜体、空柜等类型
- 有告警的机柜必须红色高亮

- 支持点击机柜聚焦
- 点击机柜后能查看单个机柜详细情况
- 支持领导展示模式，隐藏复杂编辑控件，突出整体状态和告警

- 不做复杂真实机柜门模型
- 不做复杂 GLB 资产依赖
- 不做单台服务器复杂 3D 抽屉模型
- 不做 3D 线缆拓扑
- 不为了炫技牺牲 2D 机柜图和资产管理进度

# 8. Excel 导入约束

第一版 Excel 导入采用固定模板，优先包含两个 Sheet：
- 机柜清单
- 设备清单

机柜清单字段：
- 数据中心
- 机房
- 布局类型
- 微模块，可为空
- 排，可为空
- 列，可为空
- 机柜名称
- 机柜类型
- 机柜U数
- 备注

设备清单字段：
- 数据中心
- 机房
- 微模块，可为空
- 机柜名称
- 安装面
- 起始U位
- 高度U
- 设备类型
- 固定资产编号
- 计算机名
- 管理IP地址
- 业务IP地址
- 带外IP地址
- 厂商
- 物理机型号
- 用途
- 操作系统
- 责任人
- SN号
- 维保到期时间
- 物理机硬件配置
- 状态
- 备注

Excel 导入流程：
- 选择 Excel 文件
- 读取 Sheet
- 解析机柜清单和设备清单
- 执行校验
- 显示导入预览
- 用户确认
- 写入 SQLite
- 生成导入批次记录

导入必须先预览校验结果，用户确认后才能写入数据库。

导入预览需要显示：
- 总行数
- 可导入行数
- 错误行数
- 警告行数
- 新增机柜数量
- 新增设备数量
- 跳过重复设备数量
- U 位冲突列表
- 重复 IP 列表
- 重复资产编号列表
- 字段缺失列表

导入错误 Error，不能导入：
- 机柜不存在
- 起始 U 位为空
- 起始 U 位不在 1-48
- 高度 U 为空或小于 1
- U 位冲突
- 计算机名为空
- 同一文件内固定资产编号重复
- 同一文件内 SN 号重复

导入警告 Warning，可以导入但必须提示：
- 责任人为空
- 用途为空
- 维保到期时间为空
- 操作系统为空
- 硬件配置为空
- 业务 IP 为空
- 带外 IP 为空

第一版重复设备处理策略：
- 设备唯一识别优先级为固定资产编号、SN号、计算机名、管理IP地址
- Excel 导入默认只新增设备
- 如果识别为已存在设备，标记为重复并跳过
- 第一版不自动覆盖已有设备
- 后续版本再支持字段级更新对比和确认更新
