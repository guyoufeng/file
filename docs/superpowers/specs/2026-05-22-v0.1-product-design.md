# 泉峰AI数据中心管理平台 v0.1 产品设计

## 1. 项目定位

泉峰AI数据中心管理平台是一个本地优先、开源、可扩展的 AI 数据中心机柜资产与运维管理平台。平台不是单纯的机柜画图工具，也不是单纯的 AI 聊天工具，而是以数据中心、机房、机柜、U 位、设备资产、告警状态和 AI 查询为核心，逐步建设成公司可长期使用的专业 DCIM 平台。

v0.1 的目标是先做稳一个可用、专业、可演示的基础版本：用户可以录入或导入机柜和设备资产，在专业 2D 机柜图中查看设备位置和状态，通过 AI 助手用自然语言查询服务器位置、用途、责任人、机柜清单和告警情况，并能进行基础数据备份和审计追溯。

## 2. v0.1 成功标准

v0.1 成功的标准不是功能堆满，而是形成真实可用闭环：

- 能创建和查看多个数据中心/机房。
- 能表达 529数据中心的华为微模块结构，以及其他简单机房或单柜机房。
- 能通过表单和 Excel 模板录入设备资产。
- 能在 2D 机柜图中清楚看到机柜、U 位、设备、状态和告警。
- 能按计算机名、业务 IP、管理 IP、带外 IP、资产编号、SN、用途、责任人搜索设备。
- 能通过 AI 查询服务器在哪里、用途是什么、责任人是谁、某个机柜有哪些设备。
- AI 查询结果能显示数据来源和查询时间，并写入审计日志。
- 能通过基础告警数据让机柜图高亮异常设备。
- 能导入导出项目 JSON，支持本地备份和迁移。
- 能提供轻量但可展示的 3D 机柜布局视图。

## 3. v0.1 示例数据规模

第一版内置或提供示例数据，规模接近真实演示但不压垮开发和性能：

- 数据中心：南京、杭州、越南。
- 机房/数据中心房间：529数据中心、99数据中心、308数据中心、杭州数据中心、越南C7数据中心。
- 机柜：约 50 个。
- 设备：约 200 台。
- 告警：约 10-20 条模拟或手动告警。
- 默认机柜高度：48U。

529数据中心是 v0.1 的核心示例机房，由 2 个华为微模块组成。每个微模块 20 个柜位，布局为 2 排 x 10 列，合计 40 个柜位。其他机房不强制微模块结构，可使用简单排柜布局或单柜布局。

## 4. 页面结构

v0.1 主导航固定为：

- 机柜总览
- 资产管理
- 连线管理
- 告警中心
- 报表中心
- 系统设置

AI 助手不作为独立主导航页面，而是右上角全局入口。点击后从右侧弹出，可收起、展开、放大、缩小和最小化。

导入导出在 v0.1 不做独立中心。Excel 导入和设备清单导出放在资产管理，JSON 项目导入导出放在系统设置或相关入口。后续当导出类型复杂后，再独立成导入导出中心。

## 4.1 视觉与交互风格

v0.1 默认采用暗色专业运维控制台风格，参考用户提供的机柜图方向。界面要优先服务日常运维效率和领导展示，不做营销型首页。

整体交互原则：

- 打开应用后默认进入机柜总览。
- 顶部提供数据中心/机房选择、视图切换、搜索、领导展示模式和 AI 助手入口。
- 主导航用于切换业务模块，不把 AI 助手做成单独页面。
- 机柜图、资产列表、告警列表、AI 查询结果都要使用一致的设备分类、状态和颜色体系。
- 暗色主题是 v0.1 默认主题，浅色主题只做预留。
- 专业、清晰、稳定优先于炫技动画。

## 5. 数据中心与机房建模

底层模型保留 `DataCenter + Room` 两层：

- `DataCenter` 表示地区、园区或数据中心归属，例如南京、杭州、越南。
- `Room` 表示具体机房或数据中心房间，例如 529数据中心、99数据中心、308数据中心、杭州数据中心、越南C7数据中心。

UI 上可以简化显示为“数据中心/机房”，但底层模型不合并。

Room 支持不同布局类型：

- `micro_module`：微模块布局，适合 529数据中心。
- `simple_rows`：简单排柜布局，适合一排或少量机柜的机房。
- `single_rack`：单柜布局，适合只有一个机柜的场景。

用户在机柜总览中选择实际机房名称，不直接选择“微模块总览”或“简单机房总览”。系统根据 Room 的布局类型自动展示合适布局。

## 5.1 核心数据模型

v0.1 从一开始建立稳定的数据模型，避免后续 AI、报表、监控、导入导出反复返工。

核心模型：

- `DataCenter`：地区、园区或数据中心归属。
- `Room`：具体机房或数据中心房间，包含布局类型和默认 U 数。
- `MicroModule`：微模块，仅在微模块机房中使用，简单机房可为空。
- `Rack`：机柜、柜位或基础设施柜体。
- `DeviceCategory`：设备大类和子类型。
- `Device`：设备资产主体。
- `Alert`：告警事件。
- `AiModelConfig`：AI 模型配置。
- `AiConversation` / `AiMessage`：AI 会话和消息。
- `AuditLog`：AI 查询、人工操作和未来外部 Agent 调用审计。
- `ImportBatch` / `ImportRowIssue`：导入批次和导入校验问题。
- `AppSettings`：应用设置。

v0.1 预留模型：

- `Port`：设备端口。
- `Connection`：设备端口连接关系。
- `DeviceTemplate`：设备模板。
- `ChangeRecord`：资产变更记录。

模型关系：

- 一个 `DataCenter` 可以包含多个 `Room`。
- 一个 `Room` 可以包含多个 `MicroModule` 和多个 `Rack`。
- 一个 `Rack` 可以属于某个 `MicroModule`，也可以不属于微模块。
- 一个 `Rack` 可以包含多个 `Device`。
- 一个 `Device` 可以关联多个 `Alert`。
- AI 查询和人工操作都需要写入 `AuditLog`。
- Excel 导入需要生成 `ImportBatch`，每行错误或警告写入 `ImportRowIssue`。

v0.1 不要求把所有预留模型做完整界面，但数据库迁移和类型定义要为后续扩展保留空间。

## 6. 机柜与 U 位规则

Rack 表示机房中的一个柜位或基础设施柜体，不只表示普通服务器机柜。

Rack 类型包括：

- 服务器机柜
- 网络机柜
- 存储机柜
- 配线机柜
- 列头柜
- 精密空调
- UPS/PDU
- 空柜
- 其他

精密空调、列头柜、配线机柜等特殊柜体默认可以作为整柜块展示，也要支持切换为 48U 设备视图，方便后续真实资产管理。

U 位规则：

- 默认 48U。
- 1U 在底部，48U 在顶部。
- `startU` 表示设备底部起始 U 位。
- `endU = startU + heightU - 1`。
- 例如 `startU=25`、`heightU=4`，设备占用 `25U-28U`。
- 设备不得超出 1U-48U。
- 同一机柜、同一安装面内不得出现 U 位冲突。
- v0.1 安装面支持正面、背面，2D 展示优先实现正面。

机柜名称按现场实际名称录入，平台不强制固定命名规则。

## 7. 2D 机柜图设计

2D 机柜图是 v0.1 的核心能力。

机柜总览支持三个视图：

- 布局总览
- U位大图
- 3D轻量视图

布局总览按实际机房展示。529数据中心展示 2 个华为微模块，每个微模块 2 排 x 10 列。简单机房展示一排机柜、少量机柜或单柜布局。

U 位大图横向展示多个完整 48U 机柜，风格参考用户提供的暗色专业机柜图。设备块按 U 位准确占位，并按类型和状态显示。

设备块默认显示：

- 计算机名
- 业务IP地址
- 用途

当设备块高度不足时，优先显示计算机名和业务 IP。完整字段放到悬浮提示或右侧详情面板。

缩放规则：

- 支持 50%、75%、100%、125%、150%。
- 缩小时能看到完整 48U 或更多机柜。
- 放大时能看清设备文字细节。
- 缩小时允许隐藏次要文字，避免文字混乱。
- 搜索命中设备在任意缩放级别都必须可见并高亮。
- 缩放不改变 U 位占用关系。

状态显示规则：

- 正常：按设备大类颜色展示。
- 告警：红色边框或红色角标。
- 离线：灰化。
- 维护：蓝色或紫色角标。
- 停用：暗灰。
- 备用：淡色或绿色虚线。
- 待上架：虚线占位。
- 待下架：橙色角标。
- 报废：深灰低亮度。

## 8. 3D 轻量视图

3D 视图 v0.1 定位为“轻量但具备领导展示价值”。

必须做到：

- 展示当前机房整体机柜布局。
- 能看出 529数据中心的华为微模块结构。
- 其他机房能展示简单机柜布局。
- 显示机柜编号或名称。
- 用颜色区分普通机柜、特殊柜体、空柜等类型。
- 有告警的机柜红色高亮。
- 支持旋转、缩放、平移。
- 支持点击机柜聚焦。
- 点击机柜后能查看单柜详情。
- 支持领导展示模式。

v0.1 不做复杂真实机柜门模型、复杂 GLB 资产依赖、单台服务器 3D 抽屉、3D 线缆拓扑。

## 9. 领导展示模式

v0.1 支持简单的领导展示模式，作为 2D 和 3D 的展示增强。

开启后：

- 隐藏新增、编辑、删除、导入等操作按钮。
- 突出设备总数、告警数、离线数、维保即将到期数。
- 机柜图突出告警状态和容量状态。
- 3D 视图突出整体布局、机柜编号和告警高亮。
- AI 助手可以生成当前页面摘要。
- 页面更适合投屏和汇报。

v0.1 不做专门大屏系统、轮播大屏或复杂权限。

## 10. 设备资产设计

设备分类采用“大类 + 子类型”结构。大类用于颜色、筛选、统计和报表，子类型用于资产精细分类和 AI 查询。

大类包括：

- 服务器
- 网络设备
- 安全设备
- 存储设备
- 基础设施
- 配线设备
- 其他

服务器子类型包括：物理服务器、数据库服务器、虚拟化服务器、超融合服务器。

默认子类型还包括：

- 网络设备：核心交换机、汇聚交换机、接入交换机、路由器、无线控制器。
- 安全设备：防火墙、负载均衡、入侵防御、上网行为管理、VPN设备、堡垒机。
- 存储设备：SAN存储、NAS存储、备份一体机、磁带库、分布式存储。
- 基础设施：UPS、PDU、列头柜、精密空调、KVM、环境监控。
- 配线设备：光纤配线架、网络配线架、ODF、MDF、理线架。

默认颜色体系：

- 服务器：蓝色。
- 网络设备：青绿色。
- 安全设备：红色或玫红。
- 存储设备：橙色。
- 基础设施：紫色。
- 配线设备：黄色。
- 其他：灰色。

设备状态包括：

- 正常
- 告警
- 离线
- 维护
- 停用
- 备用
- 待上架
- 待下架
- 报废

第一版物理服务器核心字段：

- 固定资产编号 `assetNo`
- 计算机名 `hostname`
- 管理/IP地址 `managementIp`
- 业务IP地址 `businessIps`
- 带外IP地址 `oobIp`
- 厂商 `vendor`
- 物理机型号 `model`
- 用途 `purpose`
- 操作系统 `operatingSystem`
- 责任人 `owner`
- SN号 `serialNumber`
- 维保到期时间 `warrantyExpireAt`
- 物理机硬件配置 `hardwareSpec`
- 所属机房 `roomId`
- 所属微模块 `moduleId`
- 所属机柜 `rackId`
- 安装面 `side`
- 起始U位 `startU`
- 高度U `heightU`
- 状态 `status`
- 功率 `powerW`
- 备注 `remark`

Excel 导入采用固定模板，包含两个 Sheet：

- 机柜清单
- 设备清单

机柜清单字段：

- 数据中心
- 机房
- 布局类型
- 微模块，可为空
- 排，可为空
- 列，可为空
- 机柜名称
- 机柜类型
- 机柜U数
- 备注

设备清单字段：

- 数据中心
- 机房
- 微模块，可为空
- 机柜名称
- 安装面
- 起始U位
- 高度U
- 设备大类
- 设备子类型
- 固定资产编号
- 计算机名
- 管理IP地址
- 业务IP地址
- 带外IP地址
- 厂商
- 物理机型号
- 用途
- 操作系统
- 责任人
- SN号
- 维保到期时间
- 物理机硬件配置
- 状态
- 备注

为了兼容早期模板，导入器可以识别旧字段“设备类型”，但系统内部应拆分为设备大类和设备子类型。无法自动拆分时，在导入预览中提示用户修正。

导入流程：

1. 选择 Excel 文件。
2. 读取 Sheet。
3. 解析机柜和设备。
4. 校验数据。
5. 显示导入预览。
6. 用户确认。
7. 写入 SQLite。
8. 生成导入批次记录。

导入必须先预览校验结果，用户确认后才写入数据库。

错误行不能导入，例如机柜不存在、U 位范围错误、U 位冲突、计算机名为空、资产编号或 SN 重复。警告行可以导入，例如责任人为空、用途为空、维保到期时间为空、业务 IP 为空。

重复设备处理策略：按固定资产编号、SN号、计算机名、管理 IP 的优先级识别。v0.1 默认只新增设备，识别为已存在设备时标记为重复并跳过，不自动覆盖。

## 15. AI 助手

AI 助手是全局能力，不作为独立页面。通过右上角入口唤起，默认从右侧弹出，支持收起、展开、放大、缩小、最小化和固定。

v0.1 AI 只做查询型助手，不做 AI 写入、修改、删除或移动设备。

支持自然语言问题：

- 哪个服务器在哪里？
- 这台服务器用途是什么？
- 责任人是谁？
- 某个机房有哪些服务器？
- 某个机柜有哪些设备？
- 这个 IP 对应哪台机器？
- 哪些设备现在告警或离线？

第一版只读工具：

- `search_devices`
- `get_device_detail`
- `locate_device`
- `list_rack_devices`
- `list_room_devices`
- `list_alert_devices`
- `summarize_room_status`

AI 回答要求：

- 专业、结构化、可追溯。
- 涉及设备时展示计算机名、业务 IP、用途、责任人、位置、状态。
- 涉及位置时展示数据中心/机房、微模块、排、列、机柜、U 位范围。
- 每次查询回答必须显示数据来源和查询时间。
- 如果查不到数据，要明确说明没有匹配记录。

## 16. AI 模型配置

系统设置中提供 AI 模型配置。v0.1 支持多个模型配置，但同一时间只启用一个默认模型。

供应商/协议分类：

- GPUStack
- OpenAI Compatible
- DeepSeek
- Gemini
- Ollama
- vLLM
- 其他

每个模型配置包含：

- 配置名称
- 供应商/协议类型
- Base URL
- API Key，可为空
- 模型标识
- 是否启用
- 请求超时时间
- 最大输出长度
- temperature
- 备注

支持发现模型、手动填写模型、测试连接。测试连接不能携带真实资产数据。API Key 不写入项目导出文件。

AI 接入通过 `AI Gateway` 实现。OpenAI Compatible 是第一版优先适配协议，但不是唯一协议。GPUStack、DeepSeek、vLLM 可以优先按 OpenAI Compatible 兼容接口接入；Ollama 和 Gemini 需要通过供应商适配器处理模型发现、请求格式和响应格式差异。

外部模型配置页面必须提醒用户：发送到外部模型的内容可能包含内网资产信息，生产使用时应开启脱敏策略或仅使用公司内网模型。

## 17. AI 审计与操作审计

v0.1 必须预留 AI 审计功能。每次 AI 问答需要记录审计日志。

AI 审计记录至少包含：

- 时间
- 用户
- 操作类型
- 问题
- 命中的工具
- 查询条件
- 回答摘要
- 关联设备
- 关联机房/机柜
- 模型配置
- 结果状态

v0.1 暂不做多人登录，默认用户可以记录为 `local-admin` 或本机用户名。后续支持用户体系后，再把真实用户、角色和权限写入审计。

审计日志入口 v0.1 放在系统设置中的“审计日志”标签页，支持按时间、关键词、设备、机房、机柜、操作类型搜索。

后续版本如果 AI 支持新增设备、调整设备、修改配置，必须把修改前、修改后、调用来源、确认人、执行结果写入审计记录。外部 AI Agent 未来通过 API 查询或修改平台时，也必须进入同一套审计体系。

## 18. 告警中心

v0.1 告警来源：

- 手动录入
- Excel 导入
- 模拟数据

告警字段：

- 告警标题
- 告警级别
- 关联设备
- 告警来源
- 告警状态
- 开始时间
- 恢复时间
- 描述
- 处理建议
- 备注

告警级别：提示、警告、严重。

告警状态：未确认、已确认、已恢复、已关闭。

告警中心支持告警列表、筛选、详情、关联设备、点击定位设备、机柜图高亮告警设备、AI 查询告警设备。不做复杂告警规则引擎。

未来通过监控适配器接入 Prometheus、卓豪监控等系统。AI 可以定期调用监控适配器 API 查询物理机状态，并更新平台健康状态、告警状态和摘要。

## 19. 报表中心

v0.1 报表中心只做基础统计和可视化卡片，不做复杂报表系统。

至少展示：

- 设备总数
- 机柜总数
- 数据中心/机房数量
- 各设备大类数量
- 各设备状态数量
- 告警设备数量
- 离线设备数量
- 维保即将到期设备数量
- 机柜容量占用率

Excel 设备清单导出放在资产管理。PDF、Word、月度报告、AI 报告生成进入后续版本。

## 20. 连线管理

v0.1 数据库预留端口和连线模型，但界面只做基础入口和空状态。

预留模型：

- `Port`
- `Connection`
- `DeviceTemplate`
- `ChangeRecord`

v0.1 不做拖拽连线、拓扑图、端口占用图、接线表导出。设备详情中可以预留端口/连线标签页。

## 21. 系统设置

v0.1 系统设置包含：

- 审计日志
- 关于系统

## 22. 数据保存与备份

v0.1 采用本地优先数据策略：

- 默认 SQLite 本地数据库。
- 应用内操作自动保存。
- 数据库文件默认放在应用数据目录。
- 支持手动备份。
- 支持 JSON 项目导入导出。

项目 JSON 导出包含数据中心、机房、机柜、设备、告警、分类等业务数据，但不得包含 API Key 或敏感凭据。JSON 导入必须校验结构和版本，导入前提示覆盖或合并。v0.1 优先支持导入为新项目或覆盖当前示例数据。

## 23. 技术栈

固定技术栈：

- 桌面端：Tauri 2
- 前端：Vue 3 + TypeScript + Vite
- 路由：Vue Router
- 状态管理：Pinia
- UI 组件库：Naive UI
- 2D 机柜图：Konva.js
- 3D 展示：Three.js
- 本地数据库：SQLite
- Rust 数据库层：sqlx
- Excel 导入导出：前端 SheetJS
- AI 接口：AI Gateway + Provider Adapter，优先支持 OpenAI Compatible API
- 模型来源：公司 GPUStack / qwen3.6-35b，后续兼容外部模型

架构原则：

- 数据模型先行。
- 桌面优先、本地优先。
- 后续预留服务化、API 化和插件化。
- 监控、AI、导入导出、通知都按适配器/插件思路设计。
- 2D 机柜图性能和清晰度优先。

## 24.1 开发与验收流程

后续开发必须按小步提交推进。每个功能完成后至少执行对应冒烟测试，测试通过后再提交 Git。

通用流程：

1. 明确功能目标。
2. 实现最小可用功能。
3. 运行类型检查、单元测试或页面冒烟测试。
4. 修复发现的问题。
5. 提交 Git。

v0.1 关键冒烟测试：

- 应用启动：Tauri/Vite 应用能启动，默认进入机柜总览。
- 机柜总览：能切换 529数据中心、99数据中心、308数据中心、杭州数据中心、越南C7数据中心。
- 布局总览：529数据中心能显示 2 个华为微模块，其他机房能显示简单布局。
- U 位大图：能显示 48U 编号、设备块、状态颜色和搜索高亮。
- 资产管理：能新增设备、编辑设备、删除前确认。
- Excel 导入：能读取模板、显示预览、阻止错误行写入、跳过重复设备。
- AI 模型配置：能保存模型配置、发现模型或手动填写模型、测试连接。
- AI 查询：能用自然语言查询设备位置、用途、责任人、机柜设备清单，并显示数据来源和查询时间。
- 审计日志：AI 查询后能生成审计记录，并支持关键词搜索。
- 告警中心：能新增或加载模拟告警，点击告警能定位设备。
- 报表中心：统计卡片能根据示例数据展示设备、机柜、告警、离线和维保数据。
- 数据管理：能导出项目 JSON，导入前能校验版本和结构。
- 3D 轻量视图：能显示机柜阵列、机柜编号、告警高亮，点击机柜能查看详情。
